浅谈高可用

在我们的架构设计中，有很多追求的指标，其中一个很重要的就是高可用，尤其是在微服务架构，高可用就显得更加重要了，因为整个系统的可用性是整个可用性的乘积

SLA

没有度量就没有管理，那么该怎么度量系统的可用性呢，这里就涉及到一个指标SLA Service Level Agrement，这个指标可以当做我们对外部的承诺，如果某个系统对外宣称达到某个等级那他就需要提供对应的可用性，在工业界，大多会使用一次故障从发生到恢复所需要的时间来度量这个指标

系统可用性% 宕机时间/年 宕机时间/月 宕机时间/周 宕机时间/天
90% (1个9) 36.5 天 72 小时 16.8 小时 2.4 小时
99% (2个9) 3.65 天 7.20 小时 1.68 小时 14.4 分
99.9% (3个9) 8.76 小时 43.8 分 10.1 分钟 1.44 分
99.99% (4个9) 52.56 分 4.38 分 1.01 分钟 8.66 秒
99.999% (5个9) 5.26 分 25.9 秒 6.05 秒 0.87 秒

从上图可以看出，如果一个系统宣称达到3 个 9 的可用性，那么他一年内宕机时间必须小于 8.76 小时，这对于一些基础设施薄弱，靠人工来运维的团队来说简直是不可想象的

HOW TO

既然可用性这么重要，那么有没有什么理论来指导我们来实现高可用呢，答案是有的

  • replica 首先我们要有对数据和软硬件的冗余，要消除单点故障，比如系统的异地部署，异地多活就是这种解决方案
  • failover 我们要有对故障的检测和监控能力，要能快速的发现问题并快速切换
  • downgrade 我们要对整体的服务进行分级，搞清楚服务的重要性和优先级，在系统故障的时候，可以对某些非核心服务进行降级来保证核心服务的可用性，甚至可以在高峰时期拒绝低优先级服务的调用
  • defense 我们要在系统上线之前就要做好防御措施，代码的复杂度要低，要有单元测试，并且要进行全链路压测，做好系统应急预案
  • infrastructure 高可用还对我们的基础设施团队提出了很高的要求，我们的系统必须要支持灰度发布，在小流量阶段就能发现问题，然后还要支持回滚

综上，如果我们要搭建一个高可用的系统，那么我们就要有工程化的流程和文化，用工程化的思维去开发，用工程化的思维去运维，用工程化的思维去管理

go-sum 的点点滴滴

每个语言都有自己的包管理工具，就像 npm 之于 JavaScript，cargo 之于 rust，Go 也不例外，它也有自己的包管理工具，而且 Go 的包管理工具也经历了自己的发展，从开始的 go path 到 go vendor 再到后来统一天下的 Go modules，今天不讨论 Go 的包管理工具演进，也不讨论 modules 的使用，今天只聚焦go.sum

组成

go.sum 文件中每一行大概都可以概括成 <module> <version> <hash>，
module 代表依赖的路径
version 代表依赖的版本
hash 以 h1 开头的字符串，代表是第一版 hash 算法生成的校验和

正常情况下，每个依赖包版本会包含两条记录，第一条记录为该依赖包版本整体（所有文件）的哈希值，第二条记录仅表示该依赖包版本中 go.mod 文件的哈希值，如果该依赖包版本没有 go.mod 文件，则只有第一条记录。如上面的例子中，v1.0.0 表示该依赖包版本整体，而 v1.0.0/go.mod 表示该依赖包版本中 go.mod 文件

生成

如果我们在 gomod 模式下引入一个新依赖的时候，gomod 会先把包下载到本地，下载完成后会对包做 hash 运算，并同时把数据更新到 go.mod 和 go.sum 当中，值得注意的是，在更新 go.sum 文件之前，为了确保文件是真实，go 还会把本地计算的哈希值和从 GOSUMDB 服务器拿到的哈希值进行比较，如果两者不同，命令会拒绝向下执行，也不会更新 go.sum 文件

优雅的错误处理方式

Go 的开发人员对 error 处理肯定不陌生，因为在 Go 的设计哲学里面，错误就是应该被处理的而不是被忽略，以至于现在提到 Go，就会说 if err != nil，但是实际上目前 error 是比较简单的，它只是一个接口，只能存储简单的字符串，所以在实际的项目中，通常有两种方式来处理 error，一种是使用 Errorf进行拼接 newErr := fmt.Errorf("error %v, err) 但是这种会丢失原始的错误类型，不能再进行比较等操作了，还有一种是我们会自定义 error struct 来存储额外的记录

但是这种方式也不通用，因为每个项目都可能有自己的 struct，那么有没有一种更通用更简单的方式呢？答案就是 warp error

fmt.Errorf

首先可以使用fmt.Errorf(%w)格式符来嵌套 error，这样就避免引入额外的方法，非常简洁

Unwrap

对于嵌套的 error 我们可以使用 Unwrap 来解析

Is

可以使用 Is 函数来判断某个错误是否和某个特定错误相等

As

还可以通过 As 函数从错误链条中找到 target 的类型并设置

综上就是 error wrap 的一些相关函数和方法，它极大地增强了 Go 原本孱弱的 error，感兴趣的同学可以试试
